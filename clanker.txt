MinimalistHUD project notes for future Codex agents

Project structure
- `src/` contains the modded source set used for rebuilding/importing, split by SWF:
  `src/hud.swf/` for gameplay HUD sources and `src/menu.swf/` for menu sources.
- `src_vanilla/` is the full exported vanilla ActionScript reference dump, also split by SWF:
  `src_vanilla/hud.swf/` and `src_vanilla/menu.swf/`.
- When a needed class exists only in `src_vanilla/`, copy it into the matching SWF-specific path under `src/` before modifying it.

Editing rules that match how this project is being worked on
- Preserve the watch gameplay widgets unless specifically asked to remove them.
- Prefer suppressing HUD elements at widget/controller entry points like `onSetData()` instead of ripping out embedded assets.
- Keep changes scoped to `src/`. Treat `src_vanilla/` as reference only.

Notes on bluff/lure strip
- The bottom bluff/lure widget is the agency selector strip, not the circular agency meter.
- `AgencySelectorWidget` owns the frame, lines, icon, and chunk count.
- `AgencySelectorWidgetEntry` owns each individual prompt glyph, flare effect, and per-entry cost/label layout.
- Current mod behavior hides the frame and chunk count, hides entry labels and costs, and keeps only the prompt glyphs plus flares when enabled.
- The whole bluff/lure widget is currently gated by whether at least one prompt reports `m_promptData[0].isEnabled`, and it also explicitly restores the widget from its hidden state when enabled prompts return.
- Current mod behavior also disables `frame_mc` entirely, removing the frame icon, frame count, divider lines, corners, background, and residual dot artifacts, and hides the per-entry numeric cost labels, leaving a glyph-only bluff/lure strip.
- A transient Confront bug was traced to `AgencySelectorWidget` tearing down and rebuilding entries immediately when prompt count changed, even if the new prompt payload was incomplete for that frame. Current mod behavior now validates the replacement payload before rebuilding and explicitly restores widget visibility when a showable prompt set is present.

Notes on standard button prompts
- The shared interaction prompt pill/glyph renderer is `ButtonPromptWidget`.
- The small out-of-range interaction ring/circle is not a separate `InteractionIndicator` class here; it is the shared `collapsed_mc` path used by `ButtonPromptWidget` in its `STATE_AVAILABLE` and `STATE_COLLAPSED` states.
- Vanilla behavior uses `ButtonPromptImage` instances with optional per-prompt inversion and platform-specific scaling.
- Current mod behavior forces prompt images through the inverted color matrix so the buttons render dark with light glyphs, and applies a `0.8` multiplier to the normal prompt icon scale.
- `CloseCombatPromptsWidget` is separate and does not inherit those shared prompt changes automatically.
- Current mod behavior also forces its `ButtonPromptImage` instances through the inverted color matrix and applies the same `0.8` scale multiplier there.
- `NPCCombatPromptsWidget` is also separate and directly instantiates its own combat prompt images.
- Current mod behavior also forces those combat prompt images through the inverted color matrix and applies the same `0.8` scale multiplier there.
- `ButtonPromptsWrapper` is another direct `ButtonPromptImage` user, used by grouped melee-finisher style prompts.
- Current mod behavior now also forces those wrapped prompt images through the inverted color matrix.
- Takedown-style combat prompt title text in `NPCCombatPromptsWidget` is now hidden entirely instead of recolored.
- The `Shift + LMB` style slam-into-enemy prompt appears to be `TYPE_COMBAT_RUSH` in `NPCCombatPromptsWidget`; current mod behavior suppresses that prompt at the `onSetData()` gate.
- The chemical/electrical resource side icons in `ButtonPromptWidget` were a separate path from the right-side pickup popup controller.
- Current mod behavior now suppresses those shared prompt resource icons whenever they would be shown in the disabled/blocked state and `SHOW_PICKUP_NOTIFICATIONS` is `false`, and also fixes a stale-state bug where `showResourceType()` did not refresh when the icon stayed conceptually "shown" but its resource subtype or disabled state changed.

Notes on direction arrows
- `AttentionGainWidget` was already overridden and now hard-hides the wedge-based attention indicators in `onSetData()`.
- Remaining red directional combat arrows were traced to the damagedirection controllers, not the attentiongain widget.
- `OffscreenEnemyDirectionWidget` colors a container with `MenuConstantsKnt.COLOR_HUD_DANGER_HIGH` and shows per-enemy indicators when `newCombatant` is true.

- Current mod behavior ports both damagedirection classes into `src/` and keeps their vanilla object/state lifecycle intact while forcing the rendered indicators hidden. This is safer than fully short-circuiting `onSetData()` if combat stability depends on normal widget bookkeeping.

Notes on objectives widget
- The top-left mission/objectives list is controlled by `ObjectivesWidget`.
- `ObjectivesWidget` already had watch-state handling through `IsInQLens` / `m_isAimingWatch`.
- Current mod behavior keeps the objectives list hidden unless the watch/QLens is being aimed, and scales the whole widget to `0.85`.
- The centered tutorial-style banner is not the same widget; tutorial center-banner notifications also flow through `ObjectivesNotificationItemWidget` using `ObjectivesData.TYPE_TUTORIAL`.
- Current mod behavior suppresses only the `TYPE_TUTORIAL` branch there, leaving item, challenge, mosaic, and collectible notifications intact.
- A separate center notification family also exists in `ObjectivesNotificationWidget` and `ObjectivesHintRevealingWidget`.
- `ObjectivesNotificationWidget` is the one with the shrinking progress bar / countdown fill for untracked opportunities.
- `ObjectivesNotificationItemWidget` is another center notification path and is also the widget that explicitly handles `ObjectivesData.TYPE_TUTORIAL`.
- Current mod behavior no longer suppresses that tutorial path there; it now allows it to render at `0.5` scale along with the other remaining center objective/hint/tutorial banners.
- The world-space objective marker with the distance text is `ObjectivesMarkerWidget`.
- Current mod behavior ports `ObjectivesMarkerWidget` into `src/` and hides it unless watch mode is active. Because the marker payload itself does not reliably carry `IsInQLens`, `ObjectivesWidget` now publishes its watch state through `ObjectivesMarkerWidget.setGlobalWatchState(...)`, and marker instances refresh immediately from that shared flag instead of waiting for the next movement/camera-driven marker update.

Notes on eavesdrop HUD
- The eavesdrop HUD is controlled by `knt.hud.eavesdrop.EavesdropWidget`.
- Current mod behavior ports `EavesdropWidget` into `src/` and keeps its internal state machine running, but gates the root widget on the shared Q-watch/Q-lens state so it only appears while watch mode is active.
- `ObjectivesWidget` also publishes that shared watch state to `EavesdropWidget.setGlobalWatchState(...)` for immediate show/hide updates when the watch is toggled.

Notes on focus gain popup

- Vanilla behavior compares `agencyData.currentAgency` against `m_previousValue` and shows a popup when the value increases.
- Current mod behavior ports `AgencyNotificationWidget` into `src/` and suppresses the popup in `onSetData()` while still updating `m_previousValue` so repeated gains do not accumulate stale deltas.

Notes on save spinner
- The lower-right `Saving` spinner is driven by `knt.hud.saveindicator.SaveIndicatorWidget`.
- Current mod behavior ports `SaveIndicatorWidget` into `src/` and hard-hides it in both `onSetData()` and `onSetVisible()` while killing any running animation handles on its `outer_mc`, `inner_mc`, and `title_txt`.

Notes on rubber-cam look prompt
- The lower-right cutscene `Look` prompt appears to be driven by `knt.hud.buttonprompts.RubberCamPromptWidget`.
- This widget swaps its prompt glyph based on `data.prompt.controllerType`, supports keyboard glyphs or controller icons, and animates the input icon side-to-side while `data.isActive` is true.
- Current mod behavior keeps its data/layout refresh path intact but kills the prompt animation and forces the widget hidden in `onSetData()`.

Notes on melee counter / dodge indicators
- The combat counter/parry and dodge/sidestep overlay is driven by `knt.hud.melee.MeleeAttackWidget`.
- Its embedded view clips are `parry_mc`, `perfect_parry_mc`, `parry_success_mc`, and `unblockable_mc` on `MeleeAttackWidgetView`.
- Vanilla behavior animates those clips for normal/parry/perfect-parry windows plus sidestep/unblockable windows and counter success feedback.
- Current mod behavior suppresses the widget at the controller layer by killing any active melee indicator animations in `onSetData()` and keeping both the root widget and inner view hidden.
- That suppression is now controlled by the global `MenuConstantsKnt.SHOW_MELEE_ATTACK_WIDGET` flag, which currently defaults to `false`.

Notes on pickup popups
- The right-side ammo / chemical / electrical pickup popup is driven by `knt.hud.notifications.PickupNotificationWidget`.
- Vanilla behavior raises entries for `Ammo.Amount` and `Resource.Amount`, with resource type variants for ammo, electrical, and chemical pickups.
- Current mod behavior ports `PickupNotificationWidget` into `src/` and suppresses it entirely by clearing/hiding active entries in `onSetData()` instead of raising new notifications.
- That suppression is now controlled by the global `MenuConstantsKnt.SHOW_PICKUP_NOTIFICATIONS` flag, which currently defaults to `false`.
- The menu SWF also has its own duplicate `PickupNotificationWidget`; current mod behavior now mirrors the same suppression there so checkpoint reloads do not revive the disabled pickup icons through the menu-side controller path.

Notes on NPC watcher indicators
- The white overhead watcher dot/icon is driven by `NpcStateWidget` via `state_mc.watcher_mc`.
- The influenced-by-watcher icon is also driven there via `state_mc.watcher_influenced_mc`.
- Vanilla behavior already gated `INFLUENCED_BY_WATCHER` on `m_isInWatchView`, but plain `WATCHER` / `WATCHER_INFLUENCING` did not require watch view.
- Current mod behavior ports `NpcStateWidget` into `src/` and short-circuits the widget entirely unless `m_isInWatchView` is true, so all NPC overhead state indicators from this widget only appear while using Q Lens/watch view.

Notes on weapon ammo bar
- The redundant right-side bullet strip is `WeaponWidgetView.ammo_bar`, controlled by `knt.hud.weapons.WeaponWidget`.
- Primary ammo bar activation happens when a weapon loads:
  `m_view.ammo_bar.visible = true`
  `drawAmmoBarSegments(data.nMaxAmount, 142, 90, 1)`
- Segment visibility is updated through `m_ammoBarSegments[...]` based on `nRemainingAmount`.
- Related supporting visuals are `gradient_lines_mc`, `ammo_icon_container_mc`, `ammo_info_mc`, and for gadget/resource weapons `special_ammo_bar`.
- If removing only the redundant bullet strip, `ammo_bar` is the first target. Be careful not to accidentally remove the numeric ammo text in `ammo_info_mc`.
- Current mod behavior gates the primary ammo bar off in `WeaponWidget` with a `SHOW_AMMO_BAR` constant, so the numeric ammo digits remain but the curved right-side primary ammo bar is no longer shown or updated.
- The leftover arc/separator visuals next to the removed ammo bar were `gradient_lines_mc`; current mod behavior keeps those hidden as well.
- When the widget was changed from dim-on-disabled to hidden-on-disabled, using `m_view.visible = false` caused an incomplete restore path where ammo digits returned but some icon elements did not.
- Current mod behavior now hides the disabled weapon HUD via root `alpha = 0` instead of root visibility, and restores via alpha as well.
- That disabled-state alpha handling now also includes `super_special_prompt_container_mc`, which fixes the ESP discharge prompt lingering on-screen while the rest of the weapon HUD is hidden.

- Current mod behavior removes the marker icon grow/pulse animation and snaps the icon directly to its resting scale, while keeping the watch-only visibility behavior.
